# Health Check Skill

プロジェクトのコード・設定・依存関係を読み込んで、バグ・壊れ・危険な箇所を洗い出す。

## 実行手順

1. CLAUDE.md・package.json・主要なソースファイルをすべて読み込む
2. 以下のチェックリストを順番に確認する
3. 問題が見つかったら `ファイル:行番号` 形式で列挙し、深刻度と修正案を添える
4. 問題がなければ「✓ 問題なし」と記載する

## チェックリスト

### 1. クラッシュ・例外
- null / undefined アクセスで落ちそうな箇所（optional chaining ?.  が抜けている）
- 配列の範囲外アクセス（sessions[activeIdx] など、要素が0件のとき）
- JSON.parse / parseInt など例外が出うる箇所に try-catch がないか
- 非同期処理（async/await・Promise）で reject が握りつぶされていないか
- useEffect のクリーンアップ漏れ（setInterval・setTimeout・EventListener・WebSocket）

### 2. React固有
- useEffect の依存配列が不完全（eslint-disable で誤魔化している箇所）
- stale closure（古い値を参照したまま状態更新している箇所）
- コンポーネントのアンマウント後に setState している箇所（mounted フラグ確認）
- key が index だけで要素の並び替え・削除時に壊れそうな箇所
- forwardRef / useImperativeHandle の型・nullチェック漏れ

### 3. WebSocket・通信
- 接続が切れたままメッセージを送ろうとする箇所（readyState チェック漏れ）
- 再接続ループが止まらない条件がないか（バックオフなしで無限リトライ）
- メッセージハンドラが複数回登録される可能性（addEventListener の重複）
- サーバー側でクライアント切断時のリソース解放漏れ（pty.kill() 呼び忘れ）

### 4. 状態管理・データ整合性
- セッション削除・リネーム後に古い参照が残っている箇所
- localStorage の読み書きで quota exceeded / JSON 破損時の対処
- 複数タブ・複数クライアントで状態が食い違う可能性
- ポーリングと WebSocket イベントが競合して二重更新になる箇所

### 5. サーバーサイド（Node.js）
- ユーザー入力がそのままシェルコマンドに渡るコマンドインジェクション箇所
- ファイルパスのトラバーサル（../../../ 等で意図しない場所を読み書きできないか）
- 認証・セッション管理が必要なエンドポイントに保護がかかっているか
- process.env の必須変数が undefined のときクラッシュしないか
- 大量リクエスト・大きなペイロードへのレート制限・サイズ制限

### 6. 設定・環境
- package.json の依存バージョンに既知の脆弱性がないか（npm audit で確認）
- .env・シークレットがソースコードにハードコードされていないか
- Node.js バージョン互換性の問題（engines フィールドと実行環境が一致しているか）
- ビルド成果物（dist/）が gitignore されているか

### 7. エッジケース・境界値
- セッションが0件のときのUI（空状態の表示漏れ）
- セッション名に特殊文字（スペース・日本語・記号）が含まれたとき壊れないか
- ネットワークが極端に遅いとき、タイムアウト処理があるか
- 画像アップロードで巨大ファイル・不正な形式を送ったときのハンドリング

### 8. メモリ・パフォーマンス劣化
- タイマー・インターバル・イベントリスナーが積み上がって増え続ける可能性
- ターミナル出力バッファが無制限に増加しないか
- 長時間使用でメモリリークが起きそうな箇所

## 報告フォーマット

```
## [チェック項目名]
✓ 問題なし  または

🔴 [critical] `ファイル:行番号`
> 問題の説明
> 修正案: ...

🟠 [high] `ファイル:行番号`
> ...

🟡 [medium] `ファイル:行番号`
> ...

🔵 [low] `ファイル:行番号`
> ...
```

最後に「今すぐ直すべき TOP 3」を一言でまとめる。
